把数据湖文件变成可查、可写、可互操作的表

pg_lake 的核心目标不是“再造一个数据库”，而是让 PostgreSQL 继续做 SQL、事务和权限入口，让 DuckDB 快速读写对象存储文件，让 Iceberg 保存标准湖表元数据。

PostgreSQL：入口和事务 pg_lake：桥接层 DuckDB：文件执行引擎 Iceberg：表格式 对象存储：数据落地

1. 基础概念先分清

文件格式

Parquet/CSV/JSON 解决“一个文件怎么存”。Parquet 是列式格式，适合分析扫描。

表格式

Iceberg 解决“很多文件怎么组成一张表”。它管理 snapshot、manifest、schema 演进和删除语义。

执行引擎

DuckDB 负责快速读写文件，尤其适合列式扫描、filter pushdown 和批量计算。

数据库入口

PostgreSQL 接收 SQL，负责事务、权限、规划和不能下推时的本地执行。

Catalog

Catalog 把表名映射到当前 metadata 文件，并提供原子提交点。

Pushdown

把适合 DuckDB 的扫描和计算推下去，减少 PostgreSQL 一行行处理的成本。

最容易混淆的一点：Parquet 不是 Iceberg。Parquet 是“书页怎么印”，Iceberg 是“目录、版本和哪些书页属于这张表”。

表格式与存储层Iceberg metadata、manifest、对象存储保存标准湖表状态和真实数据文件

3. SELECT 查询怎么跑

规划阶段

pg_lake 用 FDW callback 和 planner hook 判断哪些 SQL 可以下推。如果全查询可下推，走 CustomScan；否则走 ForeignScan 加 PostgreSQL 本地处理。

创建扫描快照

执行开始时根据 PostgreSQL 事务快照确定当前可见的表版本，避免查询看到不一致的文件列表。

文件发现和裁剪

从本地 catalog 或 Iceberg metadata 找到当前 data/delete files，再用分区值、min/max 统计和谓词推理跳过不可能命中的文件。

改写成 DuckDB SQL

lake 表会被改写成 read_parquet(...) 等 DuckDB 函数。如果有 position delete，还会生成基于文件名和行号的 anti-join。

流式返回 PostgreSQL

pgduck_server 执行 DuckDB 查询，通过 PostgreSQL wire 协议把文本行返回，pg_lake 再转换成 PostgreSQL tuple。

SELECT * FROM lake_table WHERE dt = DATE '2026-06-26'
  -> file pruning
  -> read_parquet(candidate_files)
  -> apply position deletes if needed
  -> stream rows back to PostgreSQL

4. 写入、更新、删除

INSERT

接收行数据，暂存，再由 DuckDB COPY 成新的 Parquet data files。提交时把新增文件写入 Iceberg metadata。

DELETE / UPDATE

DELETE 记录源文件和文件内行号。UPDATE 可以理解成“删除旧行 + 插入新行”。

情况策略解释
整个文件都删除移除 data file 引用只改目录，不重写文件
删除少量行MOR / position delete写小删除文件，读时过滤
删除大量行COW / 重写文件重写干净文件，降低后续读放大
MOR 写得快，但 delete file 多了会拖慢读；所以 VACUUM/compaction 是湖表系统的核心能力。

5. Catalog 是“当前版本指针”

表名catalog.namespace.table
metadata-location当前元数据文件路径
metadata JSONschema、snapshot log
snapshot某一刻的表状态
manifest文件清单和统计
data/delete files真实数据和删除记录

pg_lake 在 PostgreSQL 本地 catalog、REST Catalog、object-store catalog 之间做桥接。最难的是提交协调：本地状态、对象存储文件、Iceberg 元数据和外部 catalog 必须尽量保持一致。

7. GaussDB 评估清单

  • 是否有等价 FDW、planner hook、executor hook 或 table access method 接入点？
  • DuckDB 作为外部进程、内嵌库，还是替换成其他向量化引擎？
  • 本地 catalog 与 Iceberg metadata 如何保持一致？
  • COMMIT、ABORT、crash recovery、REST catalog 失败如何恢复？
  • 写并发走表级串行化，还是实现 Iceberg 乐观并发重试？
  • 对象存储凭证、执行服务访问权限、审计和观测怎么设计？
  • 类型语义、时间、decimal、JSON、geometry、collation 如何兼容？
  • compaction、snapshot expire、manifest merge 是否作为一等能力建设？
